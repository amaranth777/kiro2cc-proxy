## 上下文
当前系统中，用量记录（UsageTracker）、限流日志（ThrottleLog）及请求错误（FailureLog）采用的是**请求内同步序列化与磁盘写入**的方案。这在异步框架 (Axum/Tokio) 内极为致命，当并发骤增时会导致请求线程全部卡死在 I/O 调用上；此外 RPM 频率限制记录只增不减（依赖后台查询触发懒删除），存在 OOM 风险。

## 目标 / 非目标
**目标**
- 实现无阻塞写入（Write-Behind），保证请求响应的高并发。
- 保证 RPM 计算的 $O(1)$ 复杂度与恒定内存消耗。
- 最大限度利用已有的 `tokio` 基础设施。

**非目标**
- 引入数据库（SQLite/Redis）去维护状态。由于本质仍是文件配置型代理服务，应通过良好的内存结构管理状态，而无需引入运维负担。

## 决策
1. **RPM Tracker - 环形缓冲区 (Ring Buffer)**
   - 放弃 `Vec<Instant>`。
   - 使用包含 `[Bucket; 60]`（对应一分钟的 60 秒）的定长数组。在 `record_request()` 时获取时间戳的秒数并取模作为索引，若时间戳跨度超前，则覆盖该槽并重置计数，实现了完全在常数时间与常数内存下的计数更新。

2. **统计/日志落盘 - 异步 MPSC Channel**
   - 引入 `tokio::sync::mpsc::unbounded_channel` 进行核心解耦。
   - 将各个 `Tracker` 作为发送端 (`Sender`) 派发事件。
   - 使用 `tokio::spawn` 启动独立守护任务充当接收端 (`Receiver`)。
   - 为了性能（并非每个事件都写入一次文件），接收端使用 `tokio::time::interval` 结合信道接收，实施最高频每 N 秒写一次盘的防抖 (Debounce) 机制。
   - 此决策遵守 config 约定：“阻塞操作必须放入 `tokio::task::spawn_blocking`”，针对 `fs::write` 会被抛入专用的阻塞线程池中运行。

## 风险 / 权衡
- 采用 MPSC Channel 及内存汇聚防抖的方式意味着**日志可能会丢**：在服务器意外断电或者进程被系统 OOM Killer 强杀的情况下，Channel 内尚未执行 `fs::write` 的几秒钟日志将丢失。
- 考虑到本系统的定位是成本追踪而非强事务性金融计费，用“微量未写磁盘的数据丢失”来换取“整体网关性能的数倍跃升及系统稳定”，是一个可接受且更优的架构权衡。
